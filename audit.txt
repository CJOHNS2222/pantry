SECTION 1 — RENDERING & COMPONENT ARCHITECTURE
--------------------------------------------------------------------------------

[PERF-001] [COMPLETED] CRITICAL — God Component: PantryScanner.tsx (3,596 lines)
File: components/pantry/PantryScanner.tsx
Problem:
  A single component handles barcode scanning, receipt OCR, batch editing,
  category management, search/filter, virtualization, and form state
  simultaneously. Every state change in any sub-feature triggers a re-render
  of the entire component tree rooted here.
Impact: High CPU cost on every inventory mutation; slow initial render on
  low-end Android devices (the app's primary market).
Fix:
  - Split into: <PantryHeader />, <PantryCameraCapture />, <PantryItemForm />,
    <PantryList />, <PantryBatchEditor />, <PantryCategoryManager />.
  - Apply React.memo() on each sub-component.
  - Move scanning state into usePantryScan() hook.
  - Move form state into usePantryForm() hook.

[PERF-002] [COMPLETED] CRITICAL — God Hook: useDataManagement.ts (1,997 lines)
File: hooks/useDataManagement.ts
Problem:
  A single hook owns 14+ independent domains (inventory, shopping list, meal
  plan, saved recipes, household, ratings, undo, allergy checks, food-waste
  analytics, usage limits, expiration notifications, online status) all in
  one re-render cycle. Any state change in any domain re-evaluates the
  entire hook body.
Impact: Every Firestore snapshot re-triggers expensive useMemo computations
  (consumptionSuggestions, expirationAlerts, recipeSuggestions) for ALL context
  consumers simultaneously.
Fix:
  - Split into: useInventory(), useShoppingList(), useMealPlan(),
    useSavedRecipes(), useHousehold(), useRatings(), useExpirationAlerts(),
    useFoodWaste().
  - Each hook manages only its own listener and state.

[PERF-003] [COMPLETED] HIGH — O(n^2) Loop in calculateRecipeFeasibility Inside useMemo
File: components/recipes-meals/RecipeFinder.tsx, lines 68-123
Problem:
  getRecipeSuggestions (useMemo) maps over ALL savedRecipes and for each calls
  calculateRecipeFeasibility(), which is itself a nested loop:
  recipeIngredients.forEach -> availableItems.some(). No early exit or
  memoization. For 100 recipes x 15 ingredients x 200 inventory items = 300,000
  comparisons per render.
Impact: Visible jank when switching to Recipe Finder tab with a full pantry.
Fix:
  - Pre-compute a Set<string> of normalized inventory item names once.
  - Run feasibility only for visible recipes (paginated), not all 100+.

[PERF-004] [COMPLETED] HIGH — 20+ Hardcoded CSV Recipes Defined Inline in Render Scope
File: components/recipes-meals/RecipeFinder.tsx, lines 407-600+
Problem:
  csvRecipes is an array literal of 20+ full recipe objects defined inside
  the component function body. JavaScript re-allocates this on every render.
Impact: Unnecessary heap allocations on every re-render.
Fix:
  - Move to a module-level constant outside the component, in a separate
    data/fallbackRecipes.ts file.

[PERF-005] [COMPLETED] HIGH — Fuse.js Index Rebuilt on Every Search Call
File: utils/searchUtils.ts, lines 52-72
Problem:
  searchPantryItems() and searchRecipes() create new Fuse(items, options)
  on every invocation. Fuse.js index construction is O(n) with significant
  object allocation. Runs on every keystroke.
Impact: Sluggish search on pantries with 200+ items; high GC pressure.
Fix:
  - Cache the Fuse instance; invalidate only when the items array ref changes.
  - Use a useFuseIndex(items, options) hook with useRef + useMemo.

[PERF-006] [COMPLETED] MEDIUM — JSON.stringify Deep-Clone in Firestore Snapshot Callbacks
File: hooks/useDataManagement.ts, lines 77, 105, 149
Problem:
  prevShoppingListRef.current = JSON.parse(JSON.stringify(sortedItems)) runs
  inside every Firestore snapshot callback. JSON round-trip is 10-50x slower
  than structuredClone().
Fix:
  - Replace all instances with structuredClone(sortedItems).

[PERF-007] [COMPLETED] MEDIUM — Full Array Sort on Every Firestore Snapshot
File: hooks/useDataManagement.ts, lines 73, 101
Problem:
  items.sort((a, b) => a.item.localeCompare(b.item)) runs on every snapshot,
  even when only a single item's purchased field changes.
Fix:
  - Sort at write time in ShoppingListCacheService.addItemToCache().
  - Remove sort from the read (snapshot) path entirely.

[PERF-008] [COMPLETED] MEDIUM — useMemo No-Op Wrapper for mealPlan
File: hooks/useDataManagement.ts, line 454
Problem:
  const mealPlan = useMemo(() => mealPlanState, [mealPlanState]) adds a
  useMemo with zero computation; functionally identical to a direct reference.
Fix:
  - Remove the useMemo wrapper; use mealPlanState directly.

[PERF-009] [COMPLETED] LOW — validateMealPlan() Runs on Every setMealPlan Call
File: hooks/useDataManagement.ts, lines 405-452
Problem:
  setMealPlan() re-filters and re-sorts the full array on every invocation,
  including frequent Firestore-driven updates.
Fix:
  - Validate only at user-initiated write time, not on every state setter call.

SECTION 2 — FIRESTORE READS, WRITES & REAL-TIME LISTENERS
--------------------------------------------------------------------------------

[PERF-010] [COMPLETED] CRITICAL — Analytics Event Fired on Every collection() Reference
File: services/databaseMonitoringService.ts, lines 143-146
Problem:
  DatabaseMonitoringService.collection() calls
  AnalyticsService.trackDatabaseOperation('read', path, 0) every time a
  collection reference is constructed — even for query setup that has not
  yet executed. Firebase Analytics has a 500-event-per-session cap.
Impact: Infrastructure telemetry saturates the event cap, silently dropping
  real business events (purchases, recipe saves, etc.).
Fix:
  - Remove the analytics call from collection().
  - Only track reads inside getDoc()/getDocs() after the actual I/O completes.

[PERF-011] [COMPLETED] CRITICAL — console.trace() Called Inside Production getDocs() Path
File: services/databaseMonitoringService.ts, line 233
Problem:
  console.trace() captures a full synchronous stack trace after every Firestore
  query. This is gated on DEV but runs in all non-production builds.
Impact: Up to 50ms added latency per query in staging/testing; masks real
  performance regressions.
Fix:
  - Remove console.trace() entirely from the production code path.

[PERF-012] [COMPLETED] HIGH — FEFO Normalization Runs Client-Side on Every Snapshot
File: hooks/useDataManagement.ts, lines 621-629
Problem:
  For every inventory snapshot the code iterates all items and for each item
  with batches, filters/sorts/extracts the earliest expiry. This is O(n x batches)
  computation on the main thread for every household member's change.
Fix:
  - Move FEFO normalization to InventoryCacheService at write time (addItemToCache,
    updateItemInCache). Store the normalized expirationDate in Firestore directly.

[PERF-013] [COMPLETED] HIGH — getRatingsForRecipe() Issues Uncached Firestore Query Per Modal
File: hooks/useDataManagement.ts, lines 1748-1781
Problem:
  A new getDocs() query fires every time a user opens a recipe modal. No caching.
  10 recipe views = 10 separate Firestore queries.
Fix:
  - Cache results in Map<recipeTitle, RecipeRating[]> with a TTL.
  - Batch-fetch ratings for all visible recipes when the tab activates.

[PERF-014] [COMPLETED] HIGH — 5th Concurrent Firestore Listener Added for Household Users
File: hooks/useDataManagement.ts, lines 658-678
Problem:
  In-household users register a 5th onSnapshot listener (personal recipe cache)
  on top of household/inventory, household/presence, shoppingList, savedRecipes,
  and mealPlan listeners. Each listener holds a persistent WebSocket channel.
Impact: Increased battery drain and memory usage on mobile devices.
Fix:
  - Embed personal recipe count in the user's Firestore document
    (users/{uid}.personalRecipeCount), updated at write time. Data arrives
    via the existing user document listener in useAuth — no 5th listener needed.

[PERF-015] [COMPLETED] MEDIUM — rebuildCommunityRatedRecipes() Loads Full Collection Client-Side
File: services/recipeService.ts, lines 257-298
Problem:
  getDocs(ratingsRef) has no query constraints. Fetches the entire recipeRatings
  collection into memory, then filters by date in JavaScript.
Fix:
  - Use server-side where('date', '>=', cutoff) query.
  - Guard with isAdmin check; this function is admin/Cloud Function territory.

[PERF-016] [COMPLETED] MEDIUM — estimateRecipeCostFromIngredients() Uses Sequential Awaits
File: services/recipeService.ts, lines 115-138
Problem:
  for (const ing of ingredients) { await getIngredientPrice(...) } — sequential.
  15 ingredients = 15 chained async calls.
Fix:
  - Use Promise.all():
    const prices = await Promise.all(ingredients.map(ing => getIngredientPrice(ing)));

[PERF-017] [COMPLETED] MEDIUM — JSON.stringify for Household Equality Check on Every Presence Ping
File: hooks/useDataManagement.ts, lines 539-559
Problem:
  JSON.stringify(merged) !== JSON.stringify(prevHouseholdRef.current) serializes
  the entire Household object on every household AND presence snapshot.
  Presence can update every few seconds.
Fix:
  - Compare only scalar fields: id, lastUpdated, member count.

  consumptionSuggestions, expirationAlerts, and recipeSuggestions are three
  independent useMemos all depending on [inventory]. All three fire on the

  - Combine into a single useMemo returning all three results.
  - Or colocate each in the specific component that needs it.

[PERF-026] [COMPLETED] HIGH — Firebase Analytics Called on Every Firestore I/O Operation
File: services/databaseMonitoringService.ts, services/analyticsService.ts
Problem:
  Every getDoc(), getDocs(), and collection() reference fires Firebase Analytics
  events. A single pantry load fires 10-15 analytics events simultaneously.
  Firebase's 500-event-per-session cap is saturated with infra telemetry.
Impact: Real business events (purchases, saves) are silently dropped.
Fix:
  - Remove analytics from all Firestore wrapper methods.
  - Use Firebase Performance Monitoring (perfTrace) for Firestore latency.
  - Keep AnalyticsService calls only at business-logic call sites.

[PERF-027] [COMPLETED] MEDIUM — Analytics Parameters Are Unvalidated (Silent Drop Risk)
File: services/analyticsService.ts, lines 461-474
Problem:
  All analytics calls use Record<string, any>. Firebase limits: 25 params,
  100-char keys, 100-char string values. Events exceeding limits are silently
  dropped — no visibility into data loss.
Fix:
  - Add a sanitizeParams() validation wrapper enforcing Firebase limits.

SECTION 6 — SCALABILITY & ARCHITECTURE
--------------------------------------------------------------------------------

[PERF-028] [COMPLETED] CRITICAL — Ingredient Storage Diverges from notes.txt Schema
File: All cache services / Firestore schema
Problem:
  notes.txt specifies a normalized schema: ingredients as a master dictionary,
  recipe_ingredients as a junction table with parsed quantity/unit/preparation.
  Current implementation stores ingredients as raw string arrays
  (recipe.ingredients: string[]). This means:
  1. No server-side quantity aggregation (notes.txt Step 1 SQL pattern unavailable).
  2. No grocery API readiness (raw strings can't be sent to Instacart).
  3. Shopping list consolidation is fully client-side string matching
     (consolidateShoppingList in appUtils.ts), which will not scale.
Impact: Top long-term scalability risk.
Recommended Direction:
  - Parse ingredients at save time using parseIngredientForShoppingList()
    (already exists in appUtils.ts).
  - Persist {name, quantity, unit, preparation} objects in Firestore.
  - Enable server-side aggregation via Cloud Functions.

[PERF-029] [COMPLETED] HIGH — Monolithic AppContext Causes Global Re-Renders
File: contexts/AppContext.tsx, contexts/AppActionsContext.tsx
Problem:
  Two monolithic context objects distribute all global state to all components.
  A shopping list item being checked off triggers re-renders in Recipe Finder,
  Meal Planner, Pantry, and Settings simultaneously.
Fix:
  - Split by domain: InventoryContext, ShoppingListContext, MealPlanContext,
    RecipesContext, HouseholdContext.
  - Apply React.memo() to all tab-level components.

[PERF-030] [COMPLETED] HIGH — mealPlanSyncInProgress is a Module-Level Boolean (Race Condition)
File: hooks/useDataManagement.ts, line 49
Problem:
  Module-level `let mealPlanSyncInProgress = false` is a race condition in
  React Strict Mode (effects run twice; both checks see false before either sets
  it to true).
Fix:
  - Replace with useRef<boolean>(false) inside the hook.

[PERF-031] [COMPLETED] MEDIUM — recipeCache Stored in Component useState (Cleared on Tab Switch)
File: components/recipes-meals/RecipeFinder.tsx, line 215
Problem:
  Cache lives in component state; destroyed every tab switch. Also triggers
  re-render on cache population, even though cache state doesn't affect output.
Fix:
  - Move to useRef (no re-render) or a module-level variable (persists across
    tab switches in the same session).

[PERF-032] [COMPLETED] MEDIUM — getEnhancedAutocompleteSuggestions() O(n) Scan Per Keystroke
File: utils/searchUtils.ts, lines 82-140
Problem:
  Full items array filter on every keystroke without debouncing at the utility
  level. 200+ comparisons per character typed.
Fix:
  - Debounce call site (300ms minimum).
  - Pre-build a lowercase name cache to avoid toLowerCase() per comparison.
