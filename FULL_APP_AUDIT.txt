SECTION 1 — CRITICAL: MEMORY LEAKS & RESOURCE MANAGEMENT
================================================================================

1A) UNTRACKED setInterval CALLS — 5 instances                           ✓ DONE
    These intervals are never cleared, causing memory leaks if the service
    module is re-initialized or the component unmounts.

    - services/cacheService.ts (~L28)
      Periodic cache cleanup interval — no clearInterval on teardown.

    - services/databaseMonitoringService.ts (~L789)
      Health-check polling interval — never cleared.

    - services/offlineDataCache.ts (~L39)
      Cache freshness check — no cleanup path.

    - services/leftoverNotificationService.ts (~L80)
      Notification schedule polling — never cleared.

    - services/priceDataCacheService.ts (~L94)
      Price cache refresh interval — no cleanup path.

    FIX: Each service should expose a cleanup/destroy function that calls
    clearInterval. Call these from the app teardown or unmount path.

1B) UNBOUNDED IN-MEMORY CACHES — 3 instances                            ✓ DONE

    - services/geminiService.ts (~L116-125)
      `requestCache` Map grows without limit. No TTL, no max-size eviction.
      Heavy Gemini usage could exhaust browser memory.

    - services/imageCacheService.ts
      Image blob cache has no TTL eviction or size cap.

    - services/writeQueueService.ts
      Internal queue state accumulates without periodic pruning.

    FIX: Add max-size (LRU) or TTL-based eviction to each cache. For
    geminiService, cap at ~100 entries and evict oldest on overflow.

1C) EVENT LISTENER LEAKS — 2 instances

    - services/writeQueueService.ts (~L279)
      Global event listener registered once, never removed.

    - services/performanceMonitoringService.ts (~L66)
      `beforeunload` listener added on init, never removed.

    FIX: Store listener references and remove them in a cleanup function.

1D) RACE CONDITIONS — 2 instances                                       ✓ DONE

    - hooks/useOfflineStatus.ts (~L49-80)
      Concurrent connectivity checks can overlap and produce stale results
      when network state toggles rapidly.

    - services/offlineQueueService.ts
      Queue processing can be triggered concurrently if multiple components
      call flush simultaneously.

    FIX: Add a mutex/lock flag to prevent concurrent execution.

1E) STALE CLOSURES — 2 instances                                        ✓ DONE

    - hooks/useDataManagement.ts
      Helper functions defined inside the hook capture stale `household`
      and `user` values if those change between renders.

    - hooks/useNotifications.ts
      `mealPlan` in the dependency array causes effect thrashing (object
      reference changes every render).

    FIX: Use refs for values accessed in callbacks, or memoize object
    references with useMemo.

1F) UNHANDLED PROMISES — 1 instance

    - services/freezerService.ts (~L23, L80)
      `getDoc()` calls not wrapped in try-catch. A Firestore network error
      will produce an unhandled promise rejection.

    FIX: Wrap in try-catch with appropriate error handling.

SECTION 2 — CRITICAL: BUNDLE & DEPENDENCY ISSUES
================================================================================

2A) firebase-admin IN WEB BUNDLE                                    [CRITICAL]
    package.json lists `firebase-admin` (v13.7.0) in `dependencies`.
    This is a Node.js-only library and should NOT be in the web bundle.
    It adds significant weight and may cause bundling errors.

    FIX: Move to `devDependencies` or remove entirely. It should only
    be used in `functions/` (which has its own package.json).

2B) LOOSE DEPENDENCY VERSIONS                                       [MEDIUM]
    - All Capacitor packages pinned to `^8.0.0` (allows minor bumps).
      Tighten to `~8.0.0` for patch-only updates.
    - Firebase `^12.6.0` allows major-version-breaking updates
      if semver isn't respected.

    FIX: Tighten version ranges: `~8.0.x` for Capacitor, `~12.6.x`
    for Firebase.

2C) CAPACITOR CONFIG DUPLICATION                                    [MEDIUM]
    Both `capacitor.config.ts` and `capacitor.config.json` exist.
    The JSON file has stale appName ("Smart Pantry" vs "Stock & Spoon").
    Capacitor resolves the TS file first, but the JSON presence is
    confusing and could cause issues with older tooling.

    FIX: Delete `capacitor.config.json`, keep only `capacitor.config.ts`.

SECTION 3 — HIGH: SECURITY & FIRESTORE RULES
================================================================================

3A) STORAGE DELETE PERMISSIONS TOO BROAD                            [HIGH]
    storage.rules: `allow delete: if request.auth != null` on pantry_images
    lets ANY authenticated user delete ANY user's pantry images.

    FIX: Restrict to uploader:
      allow delete: if request.auth.uid == resource.metadata.uploader;

3B) GROCERY PRICES — NO RATE LIMITING OR VALIDATION                 [MEDIUM]
    firestore.rules: /groceryPrices/{docId} allows any authenticated user
    to submit prices with no duplicate/spam checking, quantity limits, or
    required field validation (itemName, price).

    FIX: Add field validation:
      && request.resource.data.itemName is string
      && request.resource.data.itemName.size() > 0
      && request.resource.data.price is number
      && request.resource.data.price > 0

3C) RECIPE MODIFICATIONS — NO SIZE LIMIT                            [MEDIUM]
    firestore.rules: recipeModifications.description has no size() check.
    Could allow very large payloads to be stored.

    FIX: Add `&& request.resource.data.description.size() < 500`

3D) PRICE HISTORY DATE NOT VALIDATED                                [LOW]
    Price history entries accept any date value — no check for future dates
    or obviously invalid timestamps.

SECTION 8 — MEDIUM: CONFIGURATION & BUILD
================================================================================

8A) TSCONFIG STRICTNESS GAPS                                        [MEDIUM]
    These strictness options are disabled:
    - noUnusedLocals: false          → dead imports/variables not caught
    - noUnusedParameters: false      → unused params not caught
    - noUncheckedIndexedAccess: false → array[i] not checked for undefined

    FIX: Enable `noUnusedLocals` and `noUnusedParameters` at minimum.

8B) ESLINT RULES NOT ENFORCED                                      [MEDIUM]
    - no-console: not configured — allows 150+ console.* calls in production
    - prefer-const: set to "warn" instead of "error"
    - no-var: missing entirely

    FIX: Add `"no-console": ["error", { allow: ["warn", "error"] }]`,
    change prefer-const to "error", add no-var as "error".

8C) CHUNK SIZE WARNING LIMIT HIGH                                   [LOW]
    vite.config.ts: chunkSizeWarningLimit is 1000KB (standard is 500KB).
    Large chunks degrade initial load performance.

    FIX: Reduce to 600KB and investigate any bundles that exceed it.

8D) MISSING .env.example                                            [LOW]
    No .env.example or documentation of required environment variables.
    Developers must discover VITE_GEMINI_API_KEY and other vars by reading
    source code.

    FIX: Create .env.example listing all required VITE_ variables.

8E) PWA AUTO-UPDATE STRATEGY                                       [LOW]
    VitePWA registerType is 'autoUpdate' — silently updates without user
    consent. Could cause unexpected behavior mid-session.

    Consider: Change to 'prompt' for user-controlled updates.

8F) MISSING FIRESTORE INDEXES                                      [LOW]
    No composite index for:
    - Recipe queries on (visibility, userId) — "my recipes" filter
    - Household pantry items on (is_leftover, createdAt) — leftover views

    If these queries are used, Firestore will return errors or fall back
    to client-side filtering.
