Stock & Spoon UI/UX/Usability/Layout/Functionality Audit
Date: 2026-05-30
Scope: Static audit across UI surface files (components/**/*.tsx, pages/**/*.tsx, src/components/**/*.tsx) with deep review of core user flows: Pantry, Shopping, Meal Planner, Recipe Finder, App shell/navigation/header, Login, Recommendations.

Method
- Full file sweep over UI folders to locate interaction, accessibility, visual, and maintainability risks.
- Pattern scans for keyboard/a11y semantics, tiny text usage, console/debug traces, and interaction anti-patterns.
- Targeted manual review of high-traffic screens and containers.

================================================================================
HIGH PRIORITY FINDINGS (fix first)
================================================================================

1) Unstable React keys can cause remounts and state loss in recipe cards
- Evidence: components/RecipeFinder.tsx:1456 uses titleKey with Math.random() in render path.
- Why this matters: Random keys break React reconciliation, can reset child component state (ratings/animations/focus), and degrade scroll/render performance.
- Recommendation: Use stable IDs (recipe.id || recipe.title) and avoid any randomization in key generation.

2) Collapsible header in UsageIndicator is mouse-only clickable div
- Evidence: components/UsageIndicator.tsx:108 uses role="button" on a div with onClick, but no keyboard handler.
- Why this matters: Keyboard and assistive tech users may not reliably trigger expand/collapse.
- Recommendation: Use a real button element or add Enter/Space handling and tab semantics.

3) Main content uses hardcoded viewport-height calculation that can break on mobile browsers
- Evidence: components/layout/MainContent.tsx:121 has inline height calc with 100vh and fixed subtraction.
- Why this matters: Dynamic browser chrome and virtual keyboard can create clipped content, double scroll, or hidden controls on iOS/Android.
- Recommendation: Use dynamic viewport units (dvh/svh), remove fixed subtraction where possible, and rely on flex layout with min-h + safe-area padding.

4) Very large “god components” create high regression risk and slower UX iteration
- Evidence:
  - components/PantryScanner.tsx (2993 lines)
  - components/Settings.tsx (2385 lines)
  - components/RecipeFinder.tsx (2122 lines)
  - components/MealPlanner.tsx (2033 lines)
  - components/RecipeModal.tsx (1309 lines)
  - components/ShoppingList.tsx (1047 lines)
- Why this matters: Hard to reason about state flows, more bugs, harder accessibility consistency, slower PR review and QA.
- Recommendation: Extract by feature slices (filters, lists, cards, modals, toolbars, hooks) and add Storybook-like isolated rendering/tests for each slice.

5) Core inventory list row click target is non-semantic for keyboard use
- Evidence: components/PantryScanner.tsx:1703 row div handles onClick to open detail.
- Why this matters: Keyboard users may not discover or activate row actions consistently.
- Recommendation: Make row a button/anchor or provide explicit primary action button with proper tab order.

================================================================================
MEDIUM PRIORITY FINDINGS
================================================================================

6) Excessive tiny text across major flows hurts readability/accessibility
- Evidence (sample):
  - components/EnhancedShoppingListItem.tsx:171+
  - components/ItemDetailModal.tsx:372+
  - components/MealPlanner.tsx:342+
  - components/layout/AppHeader.tsx:267+
  - components/Login.tsx:292+
- Why this matters: Small text (< 12px equivalent) is difficult for low-vision users and on small screens.
- Recommendation: Raise floor typography for body/meta text; reserve tiny text for secondary badges only.

7) Legacy console logging still present in production UI components
- Evidence:
  - components/PendingNotifications.tsx:83,171,188,202,220
  - components/PantryScanner.tsx:721
  - components/QuickAddModal.tsx:132,161
  - components/LeftoverQuickCapture.tsx:83
  - components/RecipeCommunityInsights.tsx:71,89
  - components/LeftoverAnalytics.tsx:48
  - components/RecipeRating.tsx:179,204
  - components/RecipeFinder.tsx:1361,1371,1379
- Why this matters: Noisy logs, potential PII leakage risk over time, inconsistent error handling UX.
- Recommendation: Replace with structured log service + user-safe toasts; remove debug console statements from shipped UI.

8) Notification panel and activity overlays rely on timed auto-close patterns
- Evidence: components/layout/AppHeader.tsx has timer-driven notification panel behavior.
- Why this matters: Auto-dismiss can interrupt reading or interacting, especially on accessibility devices.
- Recommendation: Keep auto-close optional, pause when focused/hovered/touched (partially done), and offer explicit persistent mode.

9) Modal stack complexity in MealPlanner can create interaction collisions
- Evidence:
  - components/MealPlanner.tsx:1946,2036,2071 multiple fixed overlays
  - components/MealPlanner.tsx:2054/2055 nested stopPropagation modal wrappers
- Why this matters: Overlapping z-index/focus traps can break back button expectations and escape behavior.
- Recommendation: Centralize modal layer manager with consistent focus trap + escape + backdrop close policy.

10) RecipeFinder mixes multiple visual card patterns for similar entities
- Evidence: components/RecipeFinder.tsx has compact card, full card, tile card with differing affordances.
- Why this matters: Inconsistent mental model and action discoverability.
- Recommendation: Standardize one primary card pattern + variant tokens; keep action placement consistent.

11) Landing page visual system diverges significantly from app theme tokens
- Evidence: src/components/LandingPage.tsx uses custom color/font stack and heavy emoji icon cards.
- Why this matters: Brand continuity gap between marketing and in-app product can reduce trust and polish.
- Recommendation: Align with shared design tokens or intentionally define a documented marketing theme system.

12) Some clickable containers are still div-based with cursor-pointer patterns
- Evidence (sample): components/RecipeFinder.tsx:1468,1516 and other card containers.
- Why this matters: Increases chance of missed keyboard semantics and inconsistent focus rings.
- Recommendation: Prefer semantic buttons/links for primary click actions.

13) Repeated inline style/z-index values reduce maintainability
- Evidence:
  - components/MealPlanner.tsx:2054 style zIndex 99999
  - components/RecipeModal.tsx:714 style zIndex 99999
- Why this matters: Hard to reason about stacking context and modal layering.
- Recommendation: Centralize z-index scale in theme constants/classes.

14) AppNavigation uses tab semantics without a matching tabpanel pattern
- Evidence: components/layout/AppNavigation.tsx uses role="tab" for nav buttons.
- Why this matters: ARIA mismatch can confuse assistive technologies if not implemented with full tablist/tab/tabpanel semantics.
- Recommendation: Use nav/links semantics for app navigation or implement full WAI-ARIA tab pattern.

15) Recommendation and search badges are improving but not yet systematized
- Evidence: recent additions in RecipeRecommendations and RecipeFinder are component-local.
- Why this matters: Badge meaning/colors may drift across screens.
- Recommendation: Create shared badge component + semantic variants (positive/warning/critical/info).

================================================================================
LOW PRIORITY / POLISH OPPORTUNITIES
================================================================================

16) Loading states vary heavily by screen
- Evidence: custom loaders in MainContent, RecipeFinder, PantryScanner, etc.
- Improvement: Standardize skeleton/loading components and loading copy tone.

17) Empty-state guidance quality is inconsistent
- Evidence: good examples exist (RecipeFinder suggestions), but other screens use generic “No data” text.
- Improvement: Add contextual CTA in each empty state (what to do next).

18) Onboarding and tutorial experiences are broad but could be progressive
- Evidence: ModernOnboarding + contextual tips + feature discovery coexist.
- Improvement: Trigger based on behavior milestones to reduce cognitive load.

19) Notification action density can overwhelm on small screens
- Evidence: AppHeader notification row includes multiple small action icons/buttons.
- Improvement: Move secondary actions into overflow menu per notification.

20) Shopping and meal planning cards could use stronger visual hierarchy
- Evidence: heavy metadata in small text with similar color/weight.
- Improvement: Increase contrast and reserve accent color for highest-priority decision info.

================================================================================
FUNCTIONALITY/UX RISK NOTES
================================================================================

21) Potential duplicate IDs generated from Math.random in lists/services can collide in extreme cases
- Evidence: components/ShoppingList.tsx:355,530.
- Recommendation: Use deterministic UUID helper (crypto.randomUUID where available) or backend-generated IDs.

22) Search, filter, and preference logic now exists in multiple places
- Evidence: RecipeFinder + MealPlanner + Recommendation service.
- Recommendation: Continue consolidation into shared utility/hook to prevent future behavior drift.

23) Multiple independent state machines in large components increase subtle bug risk
- Evidence: PantryScanner/MealPlanner/RecipeFinder each contain many modals and UX modes.
- Recommendation: Introduce reducer/statechart for modal and mode transitions in each major screen.

C) Architecture Pass (2-4 sprints incremental)
- Split top 5 largest components into feature subcomponents + hooks.
- Add test harness per subcomponent (render + interaction + a11y smoke).

D) Reliability Pass (1 sprint)
- Remove residual console.* usage from UI.
- Centralize z-index and modal stack behavior.
- Replace random-key patterns and random ID generation in render paths.
