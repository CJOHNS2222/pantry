[CODE-5] USDA API KEY IN CLIENT BUNDLE (LOW / ACCEPTABLE)
  - The USDA FoodData Central API key (VITE_USDA_API_KEY) is embedded in the
    client-side bundle (VITE_ prefix = exposed to browser).
  - The USDA API is free with generous rate limits, so the risk is low, but
    it could be abused.
  - Mitigation: Rate limit the nutrition lookup client-side (already cached
    for 90 days) and monitor usage in USDA dashboard.

  - GUEST_ITEM_CAP = 20 and GUEST_SHOPPING_CAP = 30 are defined in
    useDataManagement.ts but need to be verified as actually enforced before
    adding items (not just declared).
  - If enforcement only happens silently with no error/warning, guest users
    will be confused when items stop saving.

[CODE-9] ✅ DONE — QuickAddModal voice error handler now shows toast for real
  failures; ignores no-speech and aborted errors.

[CODE-10] RECEIPT OCR — DUAL SERVICE RELATIONSHIP (MEDIUM)
  - PantryScanner.tsx imports both receiptOcrService and pantryService
    (analyzeReceiptImage from geminiService).
  - It is not clear from the code which path is primary and which is fallback.

5. PERFORMANCE OBSERVATIONS
================================================================================

[PERF-1] PANTRY LIST VIRTUALIZATION THRESHOLD
  - PantryScanner.tsx uses CATEGORY_VIRTUALIZE_THRESHOLD = 20 items before
    enabling react-window virtualization.
  - This is reasonable but consider whether grouped-by-category rendering
    handles large lists correctly (100+ items across 8+ categories).
  - Verify that react-window FixedSizeList is used per-category and not a
    single global list (category grouping complicates standard virtualization).

[PERF-2] ✅ DONE — getMealPlanTemplate() caches the 120-entry date/dayName array
  at module scope; only rebuilds when today's ISO date changes.

[PERF-3] NUTRITION CACHE IN LOCALSTORAGE
  - nutritionService.ts stores all nutrition data in a single localStorage
    key ('nutrition_cache') parsed as JSON on every read.
  - For users with large pantries and showNutrition enabled, this object
    could become large over time.
  - Consider pruning expired cache entries (already 90-day TTL exists —
    verify this prune actually happens on reads, not just on miss).

[PERF-4] ✅ CONFIRMED OK — FIREBASE LIMITS CACHE — 30s TTL
  - usageService.ts has a 30s in-memory cache for usage limits.
  - This is appropriate. No change needed.

[PERF-5] RECIPE SEARCH — BOTH SPOONACULAR AND GEMINI
  - When a recipe search runs, both Spoonacular and Gemini may be queried.
  - Gemini calls are batched (geminiService.ts with debounce and batch
    processing). Ensure that if Spoonacular returns good results, Gemini
    is not unnecessarily invoked for the same query.

  - sentryService.ts beforeSend updated: email is now stripped from

    the email field while preserving user id and household_id for
    session-level debugging without exposing PII).

[SEC-3] IMAGE UPLOAD — CONTENT TYPE ENFORCEMENT
  - storage.rules enforces content-type = image/* and size limits (5-10MB).
  - imageService.ts handles uploads. Confirm the client-side also validates
    file type and size before upload to provide immediate user feedback and
    reduce unnecessary network traffic.

[SEC-4] VITE_USDA_API_KEY IN CLIENT BUNDLE
  - See CODE-5 above. Acceptable risk level for a free API key with rate
    limiting, but worth monitoring.

[SEC-6] PAYMENT DATA — STRIPE INTEGRATION
